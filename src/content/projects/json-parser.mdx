## Parsing, Tokenization and Abstract Syntax Tree 🌳
Have you ever wondered: *How are text formats interpreted?* At the end, they are just a bunch of characters with some kind of predefined schema and the programming languages and tools are able to translate them into their own data structures. Take a look at this:

```typescript
const raw = '{ "name": "John", "lastname": "Doe", "age": 24 }'

const parsed = JSON.parse(raw)

console.log(parsed.name) // "John"
console.log(parsed.lastname) // "Doe"
console.log(parsed.age) // 24
```

At the beginning we have a raw string with **JSON** format, and then, after apply `JSON.parse` method we are magically able to access to the values through dot object notation. Well, my dear friend, it is not magic 🧙🏼‍♂️; there are many processes which happens behind the scenes which allows transforming raw strings into usable data by the programming languages and tools. We can resume them two core processes:

- **Tokenization:** The parser scans the raw string and converts them into **tokens**; which basically are an extended meaningful representation of the key characters and set of characters defined by the syntax of the target format and extra data like the location or type of the token.
- **Abstract Syntax Tree (AST) building**: Once raw string is translated into a set of tokens, they should be transformed in a heriarchycal data structure which determines the order of the elements and how they compose together.

Particularly, for this project I opted for perform both operations at once because JSON format is extremely simple and the practice project too; so I decided to not store any tokens and just build the data structure consuming raw characters. 

## Building from inner to outer 🎯
Now we know that we need to consume characters based on some patterns and combinations defined by the **JSON** format, and then build a heriarchical data structure which represents efficiently the parsed values. *But, how can we start create the functions to consume these characters?* Well, since JSON format is built upon a recursive structure; I decided to start building functions for consume and represent simpler expressions, and then, once foundations were solid enough, create functions for composable expressions. 

Now, which are those **simpler expressions** I mentioned before? They are the primitive types of values which are allowed by the JSON format schema definition:

- **strings**: streams of characters delimited by double quotes.
- **numbers**: integer and float number representations.
- **booleans**: `true` or `false`.
- **null**: special value which indicates emptyness.

**Composed expressions** basically are kinda groups of elements. I mean, they only **groups** primitive values in ordered or unordered ways. Particularly they are:

- **arrays**: groups values sequentially and allows accessing to data through indexes.
- **objects**: groups values non sequentiallys and allows accessing to data through `key/value` pairs.

Finally, since **Composed expressions** are built upon **primitive expressions**; we should start creating functions whose parse **strings**, **numbers**, **booleans** and **null** values particularly. Then we build the functions for parse **Composed expressions**.

## Representing parsed values in Rust
In order be able to parse different kind of types and group them into a same type, I decided to use `Enums`; which is an extremely powerful feature of Rust (and mainly functional programming languages) that groups multiple types into one representation. Think about them as some kind of *union types*. The base enum will be `JsonValue`; and it will represent all the possible JSON values. Take a look at the enum code:

```rust
#[derive(PartialEq, Debug)]
pub enum JsonValue {
    // Primitive values
    String(String),
    Number(NumberType),
    Boolean(bool),
    Null(Box<Option<JsonValue>>),

    // Composed values
    Array(Vec<JsonValue>),
    Object(HashMap<String, JsonValue>),
}
```

Notice this is a **recursive enum**; some `JsonValues` can be composed for other `JsonValues`, especifically `Arrays` and `Objects`. In the other hand `Null` is a tricky one because it should represent absence of a`JsonValue`; so we just boxed it into an `Option` wrapper and we are done.

## Parsing functions

### Parse string
```rust
use nom::{
    bytes::complete::take_until,
    character::complete::char,
    sequence::delimited,
};

pub fn parse_string<'a>() -> JsonValueParser<'a> {
    Box::new(|input: &'a str| {
        delimited(char('"'), take_until("\""), char('"'))
            .parse(input)
            .map(|(next_input, value)| (next_input, JsonValue::String(value.to_string())))
    })
}
```

Taking advantage from [Nom]() as parsers combinator library and some predefined types; we can focus directly in the body of the function; where we search for expressions with the following pattern `"any characters here"`. So we use the `delimited` parser to search the "opening double quote character", then all the characters until another double quote, and then consume the "closing double quote". Finally we run the parser and map the resulting value into a `JsonValue::String`.

### Parse array
```rust
pub fn parse_array_values<'a>() -> impl Parser<&'a str, Vec<JsonValue>, Error<&'a str>> {
    |input| separated_list0(terminated(tag(", "), consume_spaces()), parse_value()).parse(input)
}

pub fn parse_array<'a>() -> JsonValueParser<'a> {
    Box::new(|input| {
        delimited(
            terminated(char('['), consume_spaces()),
            parse_array_values(),
            preceded(consume_spaces(), char(']')),
        )
        .parse(input)
        .map(|(next_input, arr)| (next_input, JsonValue::Array(arr)))
    })
}
```

This function is a little bit complex; **but let's go ahead**:

First of all, we have a helper function `parse_array_values`; which returns an array of parses a `JsonValue` and then checks if exists a comma character and consume it with the incoming whitespaces. In this way, we finally get an array of `JsonValues`.

The other function is similar to the `parse_string` function described previously; we look for an "opening square bracket", then use the helper function to parse all internal elements as `JsonValues`, and finally look for a "closing square bracket". If all the parsing process is successful, then map the array of values into its enum representation as `JsonValue::Array`.
